### FireSim_URP — FirstPersonController 死代码清理 + 功能增强

**目标文件**: `Assets/Scripts/Author/Wt/FirstPersonController.cs`

**删除的死代码**：
- `PlayerStepRespond` 整个方法（~44行）
- `HandleFireExtinguisher` 整个方法 + `canSpray` / `_isSpraying` 字段
- `Init()` 空方法
- `customParticalExtinguish` 注释字段

**新增：Alt 键光标解锁**
```
按住 Alt → Cursor.Confined + visible → 操作 UI
松开 Alt → LockCursor(true) → Locked + invisible
```
逻辑在 `Update()` 最前面，按住 Alt 时直接 return 跳过视角和移动。

**🐛 Bug 修复 — 松开 Alt 光标不可见**

初始实现中只改了 `Cursor.lockState`，没有同步更新 `_isCursorLocked` 标记。松 Alt 时判断 `!_isCursorLocked` 永远为 false，恢复逻辑无法触发。

**修正方案**：引入独立的 `_cursorAltOverride` bool 标记：
```cs
// Update() 开头
bool altHeld = Input.GetKey(KeyCode.LeftAlt) || Input.GetKey(KeyCode.RightAlt);
if (altHeld) {
    Cursor.lockState = CursorLockMode.Confined;
    Cursor.visible = true;
    _cursorAltOverride = true;
    return;
}
if (_cursorAltOverride) {
    _cursorAltOverride = false;
    LockCursor(true);  // → Locked + invisible
}
```

**经验**：当 Cursor 状态受多个逻辑控制时，用独立标记追踪覆盖来源，避免和原有锁定机制的状态冲突。

**新增：右键灭火器烟雾特效**
```cs
private void HandleFireSmoke() {
    if (fireSmoke == null) return;
    if (Input.GetMouseButtonDown(1)) fireSmoke.Play();
    else if (Input.GetMouseButtonUp(1)) fireSmoke.Stop();
}
```
仅在 `_isCursorLocked` 状态下触发，空引用做了保护。

### FireSim_URP — FlammableObject 火焰不燃排障

**现象**：ModelScene 中可燃物配置了 `setOnFire=true`，场景里看不到火焰。

**排障过程**（通过 Unity MCP 反射对比两个项目）：

| 对比项 | TestScene（正常） | ModelScene（异常） |
|---|---|---|
| FlameEngine.fireVFXVariant | Wild | Wild ✅ |
| FlameEngine.globalVFXMult | 1 | 1 ✅ |
| FlammableObject.setThisOnFireOnStart | True | True ✅ |
| FlammableObject 组件 | FlammableObject + BoxCollider | FlammableObject **缺 BoxCollider** ❌ |
| MeshRenderer | 有 | 有 ✅ |

**操作**：通过 MCP execute_code 批量给 ModelScene 中 4 个缺 Collider 的 Cube 添加 BoxCollider。

**最终根因**：用户移动 Main Camera 后火焰可见。真实原因不是 Collider，而是 **Frustum Culling**——火焰粒子虽然生成了，但不在相机视锥体内所以不渲染。Collider 缺失是次要因素。

**踩坑经验**：OAVA Flame 的可燃物检查清单排查顺序应为 `定位相机 → Collider → FlameEngine → FlammableObject 参数`。先确认相机能看到燃烧位置再排查其他。

### 跨项目 FlammableObject 对比诊断工作流

通过 Unity MCP 的 `unity_reflect` + `execute_code` 组合，可以无侵入对比同一类组件在两个项目/场景中的配置差异：

1. `unity_reflect(action="get_type")` 获取类型成员
2. `execute_code` 批量读取运行时字段值
3. 对比输出，定位差异项

该方法同样适用于任意 MonoBehaviour 组件的跨项目配置对比。
