# 2026-05-29 星期五

## ScoutPanel — 代码审查修复与重构

### 修复：btn_CloseLighting 空检查导致下游注册被跳过

昨天审查发现 `btn_CloseLighting` 的 null 检查过早 return，导致 `btn_Evacuate` 等后续按钮的监听器无法注册。

**改前：**

```cs
public void Init()
{
    if (btn_CloseLighting == null) return;  // ← 太早！btn_Evacuate 还没注册
    btn_CloseLighting.onClick.AddListener(...);
    btn_Evacuate.onClick.AddListener(...);    // ← 永远不会执行
}
```

**修复方案：** 用 `?.` null-conditional operator 逐按钮独立处理：

```cs
public void Init()
{
    btn_confirm?.onClick.AddListener(() =>
        StepManager.Instance.ChooseBranchByCorrectness("02", true));

    btn_CloseLighting?.onClick.AddListener(() =>
    {
        btn_CloseLighting.gameObject.SetActive(false);
        StepManager.Instance.ChooseBranchByCorrectness("04", true);
    });

    btn_Evacuate?.onClick.AddListener(() =>
    {
        btn_Evacuate.gameObject.SetActive(false);
        StepManager.Instance.ChooseBranchByCorrectness("05", true);
    });

    btn_Report?.onClick.AddListener(() =>
    {
        btn_Report.gameObject.SetActive(false);
        StepManager.Instance.ChooseBranchByCorrectness("07", true);
    });

    btn_ConfirmInstruction?.onClick.AddListener(() =>
    {
        btn_ConfirmInstruction.gameObject.SetActive(false);
        StepManager.Instance.ChooseBranchByCorrectness("08", true);
    });
}
```

**设计要点：** 五个按钮的注册逻辑互不依赖，任何一个缺失不应影响其他按钮。用 `?.` 比集中 null 检查更合理——每个按钮独立负责自己的"能注册就注册，没有就算了"。

### ChooseBranchByCorrectness 迁移到显式 stepId

配合昨天的 StepManager 重构，将所有 `ChooseBranchByCorrectness` 调用从隐式版本迁移到显式 `stepId` 版本：

| 按钮 | stepId | 语义 |
|---|---|---|
| btn_confirm | "02" | 获取场外信息 → 进入侦察 |
| btn_CloseLighting | "04" | 断电 → 进入疏散 |
| btn_Evacuate | "05" | 疏散 → 进入灭火 |
| btn_Report | "07" | 汇报 → 进入指示 |
| btn_ConfirmInstruction | "08" | 确认指示 → 排查隐患 |

### ScoutPanelStepRespond 改用 stepId 直接判断

**改前：** 通过未知的中间层判断

**改后：** 直接用 `step.stepId` 做 switch：

```cs
private void ScoutPanelStepRespond(StepAsset step)
{
    switch (step.stepId)
    {
        case "02":
            gameObject.SetActive(true);
            btn_confirm.transform.parent.gameObject.SetActive(true);
            // 只显示当前步骤对应的按钮
            break;
        case "03":
            gameObject.SetActive(false);  // 侦察阶段隐藏面板
            break;
        case "04":
            gameObject.SetActive(true);
            btn_CloseLighting.gameObject.SetActive(true);  // 只显示断电按钮
            break;
        // ... 其他步骤同理
    }
}
```

**小结：** 每个 case 分支只激活当前步骤需要的 UI 元素，其他全部隐藏。这种"白名单"式显隐控制比"隐藏几个、显示几个"的补丁式写法更不容易漏改。一条原则：**能删的就别藏，能显式控制就别依赖默认状态**。
