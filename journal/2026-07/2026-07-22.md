### 配电房项目 — EquipmentChoosePlus ItemPlusOrMinus 性能优化 + 新功能

**目标文件：** `Assets/Scripts/Author/Wt/UI/EquipChoosePlus/EquipmentChoosePlus.cs`

**问题描述：** 昨日的 `ItemPlusOrMinus` 使用 `foreach` 遍历 `equipNeededLst` 查找匹配装备，O(n) 复杂度在装备数量增多时效率低下。

**根因分析：** 每次增减操作都线性扫描列表，且嵌套了 `BuildNeededConfigDic` 构建的 Dictionary 查找表未被 `ItemPlusOrMinus` 利用。

**优化方案：** 用 `equipNeededDic.TryGetValue(equip, out ...)` 代替 foreach 遍历，O(1) 直接定位。同时新增 `AddWrongItem` 处理非法拖拽项：

```cs
internal void ItemPlusOrMinus(EquipDefinition equip, bool isAdd = true)
{
    if (equip == null || !equipNeededDic.TryGetValue(equip, out EquipNeededItem item)) return;

    item.Num += isAdd ? 1 : -1;

    if (item.Num == 0 && !IsEquipNeeded(equip))
    {
        equipNeededDic.Remove(equip);
        Destroy(item.gameObject);
    }
}
```

**新增功能：**

- **`AddWrongItem()`** — 拖拽非必需装备时动态创建错误项，显示红色状态图标，可交互删除
- **`UpdateStatus()`** — 整合三种状态的颜色和图标逻辑：必需+数量正确→白色+🟢，必需+数量错误→红色+🟡，非必需→红色+🔴
- **`SetTipsText()`** — 鼠标悬停在状态图标上时，在图标位置显示"种类正确数量正确"/"种类正确数量错误"/"种类错误"提示
- **`ChooseEnd()`** — 遍历所有已选装备校验数量，任一不符则打印 warning 并 return，全部正确才通过

```cs
internal void UpdateStatus(EquipDefinition equip)
{
    if (neededConfigDic.TryGetValue(equip, out int requiredNum))
    {
        bool isNumCorrect = equipNeededDic[equip].Num == requiredNum;
        equipNeededDic[equip].SetStatus(isNumCorrect ? sprite_Right : sprite_Error,
            default, isNumCorrect ? Color.white : Color.red);
    }
    else
    {
        equipNeededDic[equip].SetStatus(sprite_Wrong, Color.red);
    }
}
```

### 配电房项目 — UnityComponentExtention 扩展方法

**目标文件：** `Assets/Scripts/Author/Wt/UI/EquipChoosePlus/UnityComponentExtention.cs`

**新增扩展方法：**

**① `RectTransform.Contains(RectTransform)`** — 检测一个 UI 元素是否在另一个 RectTransform 区域内（用于拖拽结束的区域命中判定）：

```cs
public static bool Contains(this RectTransform container, RectTransform rt)
{
    Vector3[] containerCorners = new Vector3[4];
    container.GetWorldCorners(containerCorners);
    float width = Mathf.Abs(containerCorners[2].x - containerCorners[0].x);
    float height = Mathf.Abs(containerCorners[2].y - containerCorners[0].y);
    Rect rect = new Rect(containerCorners[0].x, containerCorners[0].y, width, height);

    Vector3[] rtCorners = new Vector3[4];
    rt.GetWorldCorners(rtCorners);
    foreach (var corner in rtCorners)
        if (!rect.Contains(corner)) return false;
    return true;
}
```

**② `Text.SetTextWithEllipsis()`** — 文本超出 RectTransform 时用 `...` 截断，利用 `TextGenerator.Populate` 计算可见字符数。

**③ `HandleTextContentFormat()`** — 将普通空格替换为 `\u00A0`（不换行空格），规避 Unity UI 自动换行在空格处截断的问题。

**踩坑记录：** `RectTransform.rect.Contains` 是在局部坐标系中判断，而拖拽检测需要世界坐标来判断 UI 元素是否重叠，因此必须用 `GetWorldCorners` 获取世界坐标四角后构造 `Rect` 再判断。

### 知识详解 — C# 哈希表（Dictionary）底层原理

**问题来源：** 灯皇日常询问哈希表知识点，对照 Java HashMap 理解 C# Dictionary 实现差异。

#### ① 什么是哈希表？

哈希表（Hash Table）是一种 **键值对（Key-Value）数据结构**，通过哈希函数将 Key 映射到数组（桶数组）的索引位置，实现平均 O(1) 的查找/插入/删除。

```
Key → [Hash Function] → HashCode → [Compression] → Bucket Index → Value
```

#### ② C# Dictionary<TKey, TValue> 内部结构

```cs
internal struct Entry
{
    public int hashCode;    // Key 的哈希码（低 31 位）
    public int next;        // 链表下一项索引（-1 表示末尾）
    public TKey key;
    public TValue value;
}

internal struct Bucket
{
    public int firstEntryIndex;  // 指向 Entry 数组的起始索引
}
```

**核心字段：**
- `int[] _buckets` — 桶数组，长度通常为质数，每个元素存该桶链表头 Entry 索引
- `Entry[] _entries` — 实际存储所有键值对，通过 `next` 串联为链表
- `int _freeList` / `int _freeCount` — 被删除 Entry 的回收链表（重用槽位）

#### ③ 插入流程

```
1. hashCode = key.GetHashCode() & 0x7FFFFFFF
2. bucketIndex = hashCode % _buckets.Length
3. 遍历 buckets[bucketIndex] 链表 → 查重（同 hashCode + Equals）
4. 分配新 Entry（优先 _freeList 回收，其次 _count++ 新槽位）
5. entry.next = _buckets[bucketIndex]; _buckets[bucketIndex] = 新索引（头插）
6. _count > 负载阈值 → 扩容
```

#### ④ 查找流程

```cs
public bool TryGetValue(TKey key, out TValue value)
{
    int hashCode = key.GetHashCode() & 0x7FFFFFFF;
    for (int i = _buckets[hashCode % _buckets.Length]; i >= 0; i = _entries[i].next)
    {
        if (_entries[i].hashCode == hashCode && _entries[i].key.Equals(key))
        {
            value = _entries[i].value; return true;
        }
    }
    value = default; return false;
}
```

#### ⑤ 哈希碰撞处理 — 分离链接法

C# 使用 **Separate Chaining（单链表）**，碰撞的 Entry 通过 `int next` 索引链接。

**vs Java HashMap：**
- Java 8+：链表 ≥8 时转红黑树（TREEIFY_THRESHOLD=8），≤6 退化为链表
- C#：始终用单链表（.NET 认为托管环境树化收益有限）
- C# 的链表是 Entry 数组内的索引链接（`int next`），**缓存亲和性更好**

#### ⑥ 扩容机制

```
触发：_count > _buckets.Length
新容量：大于当前 2 倍的最小质数（7 → 17 → 37 → 89）
操作：逐个 rehash 所有 Entry → 新 _buckets（O(n)）
```

**质数目的：** 减少取模周期性重复，降低碰撞概率。

#### ⑦ GetHashCode / Equals 规则

| 场景 | 后果 |
|:----|:-----|
| 可变 Key 的 GetHashCode() 随属性变化 | 放入后找不到 → 内存泄漏 |
| 不同对象同 hashCode 但不等 | 链表变长，退化为 O(n) |
| 只重写 GetHashCode 不重写 Equals | 引用比较，语义相等的键被视为不同 |
| GetHashCode 分布不均匀 | 某些桶过长 |

**最佳实践（不可变 Key）：**
```cs
public readonly struct EquipmentId : IEquatable<EquipmentId>
{
    public readonly int Id;
    public override int GetHashCode() => HashCode.Combine(Id, Category);
    public bool Equals(EquipmentId other) => Id == other.Id && Category == other.Category;
}
```

#### ⑧ Dictionary vs Hashtable

| 对比 | Dictionary<TKey,TValue> | Hashtable |
|:----|:------------------------|:----------|
| 类型安全 | ✅ 泛型，编译时检查 | ❌ object，运行时报错 |
| 性能 | ✅ 无装箱/拆箱 | ❌ 值类型频繁装箱 |
| Null Key | ❌ 不允许 | ✅ 允许 |
| 推荐度 | **✅ 优先使用** | ❌ 仅兼容遗留代码 |

#### ⑨ Unity 开发注意事项

- **扩容 GC Alloc**：预知容量用 `new Dictionary<>(capacity: N)`，避免在 Update 中扩容
- **枚举 Key 装箱**：Mono/IL2CPP 下枚举默认 GetHashCode 可能装箱，实现 `IEquatable<T>` 可规避
- **遍历不能修改**：用临时 List 保存待删除 Key

#### ⑩ 考点速记

| 知识点 | 一句话 |
|:-------|:-------|
| 平均时间复杂度 | O(1) 查找/插入/删除 |
| 最坏情况 | 全碰撞 → O(n) |
| 碰撞处理 | 分离链接法（单链表） |
| 扩容触发 | _count > 容量 |
| 新容量 | 2 倍最小质数 |
| C# vs Java | C# 无红黑树 |
| 线程安全 | 不安全，需 ConcurrentDictionary |
| Unity 注意 | 扩容 GC、枚举装箱 |
