# 2026-06-11 星期四

## Records

### 数字人项目 — WsTTSClient 关键 Bug 修复

**Bug 1：status=2 不缺 break 导致 TTS 循环死等**

讯飞 TTS 流式接口在发送完最后一段文本后（status=2），服务端会在回包中带上 `"status":2` 表示合成结束。但接收循环中只打印了日志，没有 break——导致循环持续运行，直到 frame 计数达到 5000 上限或 WebSocket 超时关闭。

修复：
```csharp
// WsTTSClient.cs 接收循环中
int si = fullMsg.IndexOf("\"status\":");
if (si >= 0 && int.TryParse(fullMsg.Substring(si + 9, 1), out int sv) && sv == 2)
{
    Debug.Log($"[TTS] status=2 at frame {frame}");
    break;  // ← 加上 break
}
```

修复后：1373 字文本、5 个 chunk、634 帧即退出，TTS 耗时 34.7s → 完整获取 1.78MB 音频。

**Bug 2：音频开头重复播放**

根因：WsTTSClient.SynthesizeAsync 中设了 `headStart` 分段通知机制（每累积 500KB 触发一次 `OnAudioReady`），而 Pipeline.Awake 中又注册了 `OnAudioReady → audioPlayer.PlayPCM()`。导致 headStart 触发 + final 汇总触发 → 同一个音频开头被播了多次。

修复方案：去掉 headStart 分段通知，TTS 只在完全合成完毕后通过返回值传递给 Pipeline 的统一播放逻辑。Pipeline.Awake 中也不再提前绑定 OnAudioReady 事件。

**Bug 3：音频末尾不完整**

TTS 完整返回了 1.78MB 音频数据，但 AudioStreamPlayer.PlayPCM 播放时音频末尾被截断。经过多轮调试发现是 AudioSource 播放时机和 LipSyncDriver 的 GetOutputData 竞争导致。

最终通过以下调整解决：
- Pipeline 中等待播放完成的 while 循环保持 100ms 轮询
- LipSyncDriver.Update 中增加 AudioSource.isPlaying 检查
- 确保音频数据完整传递给 AudioClip 创建

### 数字人项目 — 管线性能打点

在 DigitalHumanPipeline.AskAsync 中添加了分阶段计时：

| 阶段 | 典型耗时 |
|------|----------|
| LLM | ~10s |
| TTS 合成 | ~29-35s |
| 音频播放 | ~60-90s |
| **总计** | ~100-130s |

```csharp
var sw = Stopwatch.StartNew();
// ① LLM
(replyText, emotion) = await llmClient.SendMessageAsync(userText);
Debug.Log($"[⏱] LLM={sw.Elapsed.TotalSeconds:F1}s");
// ③ TTS
byte[] audioData = await ttsClient.SynthesizeAsync(replyText);
Debug.Log($"[⏱] TTS={sw.Elapsed.TotalSeconds:F1}s");
// ④ 播放开始
audioPlayer.PlayPCM(audioData, 16000, 1);
Debug.Log($"[⏱] Play start={sw.Elapsed.TotalSeconds:F1}s");
// ⑤ 等待播放完成
while (audioPlayer.IsPlaying) await Task.Delay(100);
Debug.Log($"[⏱] Play end={sw.Elapsed.TotalSeconds:F1}s");
```
