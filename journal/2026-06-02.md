## 配电房项目 — EmberFlicker：PerlinNoise 驱动灰烬闪烁

### 需求

还原火灾现场中**阴燃/余烬**的视觉效果——物体表面暗红色的微弱光点，亮度不规则跳动，类似烟头燃烧的效果。

### 实现

```cs
public class EmberFlicker : MonoBehaviour
{
    public float flickerSpeed = 5f;
    public float flickerAmount = 0.3f;
    
    private SpriteRenderer _sr;
    private Light _light;
    private float _baseIntensity;

    void Start()
    {
        _sr = GetComponent<SpriteRenderer>();
        _light = GetComponent<Light>();
        if (_light) _baseIntensity = _light.intensity;
    }

    void Update()
    {
        float noise = Mathf.PerlinNoise(Time.time * flickerSpeed, 0) * flickerAmount;
        if (_sr) _sr.color = new Color(1f, 0.5f + noise, 0.1f, 0.8f + noise * 0.2f);
        if (_light) _light.intensity = _baseIntensity * (0.7f + noise);
    }
}
```

### 为什么这样做

- **PerlinNoise 而不是 Random**：Random 每帧跳变会产生"闪屏"感，PerlinNoise 是连续平滑的噪声函数，相邻帧的值相近，出来的效果像**真实的火焰颤动**而不是灯泡坏掉
- **同时控制 SpriteRenderer + Light**：Sprite 管颜色（暗红→亮橙），Light 管照度——两层叠加让余烬既有"看起来的颜色"又有"照亮周围环境"的真实感
- **Alpha 也参与抖动**（0.8f + noise * 0.2f）：让灰烬时隐时现，模拟燃烧不稳定的状态
- **flickerSpeed 乘 Time.time**：用时间做 PerlinNoise 的 x 轴采样坐标，速度越快噪声变化越剧烈

## 配电房项目 — Particle System Simulation Space 的选择

### 问题

粒子系统挂在一个移动的 GameObject 上，移动时已经发射的粒子会跟着发射器一起跑，看起来粒子在做圆周运动——这显然不对。

### 根因

Particle System 的 **Simulation Space** 默认为 **Local**：所有粒子的位置、速度都是相对于发射器坐标系的。发射器移动 → 整个粒子云跟着平移。

### 解决方案

将 Simulation Space 改为 **World**：

- **World 模式下**：粒子一旦发射，其世界坐标就固定了，不受发射器后续移动影响。火焰从发射器喷出后留在原地，发射器继续前进——这才是真实物理的行为。
- **代价**：World 模式下粒子数量上限更高（因为要追踪每个粒子的世界坐标），且特定效果（如拖尾）可能需要配合其他模块。

### 选型指南

| 场景 | 推荐模式 |
|---|---|
| 固定火焰/烟雾（发射器不动） | Local（默认，性能更好） |
| 火球/投掷物（发射器移动，粒子留在原地） | World |
| 角色技能特效（跟随角色移动） | Local |
| 车辆尾气/扬尘（载具移动，粒子留路上） | World |

## 配电房项目 — Velocity over Lifetime 导致粒子跟随旋转

### 坑

Simulation Space 切到 World 后，水平移动发射器时，已发射的粒子仍然会跟随物体做诡异的圆周运动。

### 原因

**Velocity over Lifetime** 模块给粒子施加的速度是在**发射器的局部坐标系**下计算的。发射器 X 轴旋转 -90°（让火焰水平喷出），局部坐标被旋转了，粒子在 World 空间里拿到的速度方向就不对了——发射器一移动，局部坐标系跟着转，粒子速度方向实时变化，看起来就像在做圆周运动。

### 解决

**关掉 Velocity over Lifetime**。火焰的速度控制改用 Start Speed + Limit Velocity over Lifetime 等其他不依赖局部坐标的模块。

> 根本原则：World 模拟空间下，任何在局部坐标系运算的模块都可能出问题。关掉不行就换模块，别死磕。

## 配电房项目 — Particle System Collision 模块：建筑物内燃烧

### 场景需求

- 火焰朝水平方向发射（模拟火球）
- 烟雾向上升腾
- 烟雾碰到天花板后停留并平铺扩散

### 配置要点

**Collision 模块设置：**

- **Type**: World（碰撞体必须在 World 空间定义）
- **Collides with**: 指定 Layer（天花板设为专门的 Collision 层）
- **Dampen**: 0.3~0.5（碰到天花板不要完全弹飞，而是减速停滞）
- **Bounce**: 0（烟雾不需要反弹，碰到就停）
- **Lifetime Loss**: 0.2~0.3（碰撞后粒子加速消亡，模拟烟雾在天花板散开变淡）
- **Collision Quality**: Medium（天花板是平面，不需要 High 精度）
- **Send Collision Messages**: 可选开启，配合脚本做额外逻辑

**平铺效果的关键：**

烟雾碰到天花板后，粒子速度被 Dampen 大幅削减，但粒子生命周期还有剩余——它会在碰撞点附近以极低速度扩散，加上 Size over Lifetime 逐渐放大，视觉上就是**在天花板上平铺扩散**。

**双粒子系统设计：**

```
系统1: 火焰（Fire）
├── Shape: Cone，朝水平方向
├── Start Speed: 高（火球喷射）
└── Collision: 不需要（火焰就不该撞墙）

系统2: 烟雾（Smoke）
├── Shape: 与火焰同源但独立
├── Start Speed: 低 + 向上偏移
├── Velocity over Lifetime: Y 轴正向（持续上升）
├── Collision: 天花板层 → Dampen + 平铺
└── Size over Lifetime: 从小到大（扩散感）
```

## C# — partial 关键字

### 是什么

`partial` 允许**一个类/结构体/接口的定义分散在多个文件中**。

### 为什么需要它

- **代码生成场景**：Unity 的 MonoBehaviour 如果在 Inspector 中绑定字段，Unity 自动生成 `partial class` 存放序列化代码，手写逻辑在另一个文件中——互不干扰
- **多人协作**：同一个大类拆成多个文件，不同人编辑不同部分，减少合并冲突
- **大型类的组织**：把数据定义、核心逻辑、UI 回调、编辑器扩展等按职责拆到不同 partial 文件中

### 注意事项

- 所有 partial 文件必须用**相同的命名空间**、相同的**访问修饰符**
- 编译时会被**合并成一个类**，可以互相访问 private 成员
- `[SerializeField]` 字段在任何 partial 文件中都可以绑定 Inspector

## 经验小结

| 今日心得 |
|---|
| PerlinNoise 是粒子/材质抖动的首选——比 Random 更自然 |
| World Simulation Space + 发射器旋转 = 检查所有依赖局部坐标的模块 |
| 烟雾平铺天花板 = Collision Dampen + 剩余 Lifetime + Size over Lifetime |
| 复杂的火焰特效建议拆成多系统：火焰一份、烟雾一份、火花一份 |
