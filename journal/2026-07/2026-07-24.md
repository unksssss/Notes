### 配电房项目 — XRI Test 同步：装备选择系统全链路打通

今日将主项目的装备选择系统完整同步到 XRI Test 分支，核心思路：**TaskPanel → GameRoot → BackPackPanel → EquipmentChoosePlus → EquipmentMgr 数据流闭环**。

**GameRoot.cs — 全局单例接入任务装备**

```cs
// GameRoot 新增 currentTaskInfo 字段
public TaskInfo currentTaskInfo = null;

public void SetBackPackPanelActive(bool active)
{
    backPackPanel.gameObject.SetActive(active);
}
```

方案：单例持有当前任务信息，EquipmentChoosePlus 的 `Start()` 从 `GameRoot.Instance.currentTaskInfo.neededEquip` 读取需要的装备列表，按需构建 `neededConfigDic` 快速查找表。

**EquipmentChoosePlus.cs — ChooseEnd 写回 GameRoot**

```cs
private void ChooseEnd()
{
    // 逐项校验：种类错误 → 返回、数量不符 → 返回
    foreach (var kv in equipNeededDic)
    {
        var equip = kv.Key;
        int currentNum = kv.Value.Num;
        if (!neededConfigDic.TryGetValue(equip, out int requiredNum))
        {
            Debug.LogWarning($"装备种类选择错误: {equip.displayName}");
            return;
        }
        if (currentNum != requiredNum)
        {
            Debug.Log($"工具数量不符: {equip.displayName} ({currentNum}/{requiredNum})");
            return;
        }
    }
    // 校验通过 → 写回 GameRoot
    GameRoot.Instance.equips.Clear();
    foreach (var neededEquipConfig in equipNeededLst)
        GameRoot.Instance.equips.Add(neededEquipConfig.equip);
}
```

关键：选择结束时要校验每一项的种类和数量，全部通过才写回 GameRoot，否则提示不符项。

### 配电房项目 — EquipmentItemPlus XR 兼容改造

拖拽系统改造为**共享 draggingIcon + 双模式坐标定位**：

```cs
// EquipmentChoosePlus 增加共享拖拽图标
public Image img_DraggingIcon;

// EquipmentItemPlus 拖拽逻辑
private void OnDrag(PointerEventData eventData)
{
    // XR 模式：检测 worldPosition
    if (eventData.pointerCurrentRaycast.worldPosition != Vector3.zero)
        draggingIcon.transform.position = eventData.pointerCurrentRaycast.worldPosition;
    else
        draggingIcon.transform.position = eventData.position; // 鼠标模式
}
```

方案：不再用 delta 累加定位，改用 `position = eventData.position` 直接赋值（鼠标），XR 下退到 `pointerCurrentRaycast.worldPosition`。共享 draggingIcon 在 EquipmentChoosePlus 上声明，所有 Item 共用同一个。

### 配电房项目 — TaskPanel 地图交互 EventTrigger

**TaskPanel.cs — 任务地图图标 EventTrigger 三连**

```cs
// 动态创建地图上的任务图标
var taskIcon = new GameObject("taskIcon");
var img = taskIcon.AddComponent<Image>();
img.sprite = Resources.Load<Sprite>("Icon/OnFire");
img.SetNativeSize();

var trigger = taskIcon.AddComponent<EventTrigger>();
// PointerEnter → 放大
// PointerExit  → 还原
// PointerClick → 赋值信息面板
var click = new EventTrigger.Entry { eventID = EventTriggerType.PointerClick };
click.callback.AddListener((_) =>
{
    tmp_Title.text = taskInfo.taskLocation;
    tmp_OnFireCause.text = taskInfo.onFireCause.Replace("\\n", "\n");
    tmp_TaskContent.text = taskInfo.taskContent.Replace("\\n", "\n");

});
```

注意：`Replace("\\n", "\n")` — 配置中的转义字符在运行时处理为真实换行。

### 每日知识问答 — Day 8（3/5）

| 题号 | 知识点 | 结果 |
|------|--------|------|
| Q1 | Camera.main 性能陷阱 → FindGameObjectsWithTag("MainCamera") 底层 | ✅ |
| Q2 | GC 原理 → Boehm 保守式标记-清除，不压缩堆 | ✅ |
| Q3 | SRP Batcher + MPB 回退（**连续第3次失分**） | ❌ |
| Q4 | Awake/OnEnable/Start 生命周期 → 所有 Awake 完才开始 Start | ✅ |
| Q5 | WaitForSeconds + timeScale 关系（timeScale=0 时无限等待） | ❌ |

**弱点追踪：**
- **SRP Batcher + MPB 回退** — 连续3天失分！核心：`MaterialPropertyBlock` 使用后回退到非 SRP Batcher 路径，但 GPU Instancing 仍可工作。SRP Batcher 和 GPU Instancing 不互斥。
- **WaitForSeconds + timeScale** — 新增弱点。`WaitForSeconds` 受 `Time.timeScale` 影响，timeScale=0 时无限等待，需用 `WaitForSecondsRealtime` 替代。

**强化知识点：**
1. `enabled = false` 不影响已启动的协程（协程不依赖 MonoBehaviour 的 enabled）
2. `StopCoroutine` 或销毁 GameObject 不会执行 `finally` 块
3. `Camera.main` 每帧 `FindGameObjectsWithTag`，必须缓存引用
4. Unity GC 不压缩堆，产生碎片
5. 多个 MonoBehaviour 的 Awake 无序，但所有 Awake 执行完才进入 Start

## Ideas

- EquipmentChoosePlus 校验层可提取为独立 Validator，支持不同任务类型的校验规则配置
- 拖拽系统将来可抽象为 IDraggable 接口，统一鼠标/XR/触屏三种输入模式
- WaitForSecondsRealtime 的 timeScale 无关特性值得记入协程避坑模板
